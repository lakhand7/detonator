DETO·NATOR

Static scanners ask “could this be dangerous?” and drown you in false positives. detonator asks “does it actually fire?” — and proves the answer with a captured, replayable trace.

REACHABLE — a tripwire fired · exploit proven with a minimal repro
UNREACHABLE — the adversarial input ran · nothing fired

§01 The missing primitive

Reachability, earned
by execution.

Given an agent and the tools / MCP servers it uses, demonstrate whether a specific adversarial input causes that agent to take a harmful action — not by pattern-matching a description, but by running it and watching what happens on the wire.

A tool legitimately able to read files gets flagged the same as a poisoned one. Runtime gateways enforce but don’t test. Capability evals only run benign input. The gap: no one detonates the adversarial condition and records the blast.
STATIC

could be dangerous?

High recall, high false-positive. Flags by capability and by pattern-matching descriptions. Answers a hypothetical.

DYNAMIC

did it actually fire?

One binary verdict, authored by pure code over a captured log. Same input can yield opposite verdicts on two different agents — proven by a log scan.

V1 verdict is binary — REACHABLE / UNREACHABLE. POTENTIALLY_REACHABLE + LLM adjudication is deferred.

§02 Architecture — three roles

Judge out of the jury.

The orchestrator proposes and drives. The verdict is authored by pure code. The mechanism is config-based, not interception-based: we hand the agent an MCP server entry that points at the proxy — the agent connects to us because its config says so. Pure userspace, stdio subprocess.

TARGET→ detonate proxy→ detonate eval→ VERDICT ORCHESTRATOR (Claude Code) reads the verdict to decide whether to iterate — never authors it

The user supplies a runbook: how to launch it, the benign task, the path to its MCP config (so we insert the proxy), how to trigger it. No target-specific code in the repo — the orchestrator interprets the runbook at runtime. This is the agnostic axis.

Orchestrator = Claude Code
The red-teamer.
click to expand

A Claude Code skill — the only Claude-specific thing in the system. Runs the loop and authors variants: understand shape → craft variant → poison → trigger → iterate. Hard rule: it may read the verdict to decide whether to keep going; it must never declare REACHABLE/UNREACHABLE itself.

Evaluator = detonate eval
Verdict of record.
click to expand

Deterministic tripwires over the proxy log — a pure function of the log, so --replay is free and a given log always yields the same verdict. The judge is taken out of the jury: pure code decides, the LLM never does.

No container in V1. The proxy needs no netns / cgroup / root. The V1 detection surface is the proxy log only; effect-detection tiers (egress, fs_audit) that need isolation are deferred.

§03 The proxy — imperative shell, pure core

One relay. It edits exactly
one message.

A transparent, byte-faithful relay between the agent (client) and the real MCP server. It forwards every message verbatim in both directions — except the single message matched by the active inject rule. Poison only ever happens on s2c; the canary reappears on c2s. That round trip is the exploit signature.

§04 Three uniform seams

Protocol + registry + decorator.
Written once, reused thrice.

Every growth point uses the identical pattern. A Protocol is a structural contract — implementations don’t inherit or import the core, so the dependency arrow points inward and fakes are free. A decorator files each implementation into a module registry at import. New kind = write a class, tag it, ensure its module is imported. Click a kind to see it register.

Import-wiring is the one tax: a class registers only if its module runs. “My new kind won’t fire” ≈ “its module wasn’t imported.” Each seam package’s __init__.py imports its submodules so the registries populate.

§08 Scope — frozen for V1

Ruthless about the line.

Everything deferred maps cleanly to the seam it will slot into — new class, new registry entry, zero evaluator change. The discipline is the point.

In V1

  • config-based stdio proxy — record + poison, byte-faithful, JSON-RPC log transport
  • splice · description — inject strategies inject
  • canary_exfil · unauthorized_tool tripwire
  • detonate CLI — validate · config · proxy · eval · list
  • Claude Code orchestrator — the §14 loop
  • Slack worked example + hermetic replay fixtures

Non-goals

  • No container / netns / cgroup / isolation wrapper
  • No egress / fs_audit / syscall tripwires; no honeytoken files
  • No http_sse transport — stdio only
  • No overwrite / error / structured tamper strategies
  • No POTENTIALLY_REACHABLE, no LLM judge
  • No provenance / signing / VEX emitter
  • Transcript is advisory only — never a verdict source

Deferred → its seam

  • http_sse remote MITM transport
  • overwrite · error · structured inject
  • egress — off-MCP exfil tripwire
  • fs_audit · syscall tripwire
  • approval_bypass tripwire
  • POTENTIALLY_REACHABLE + LLM adjudicator verdict
  • signed RunManifest · VEX suppression output
Known, accepted V1 limitation: the proxy log only observes harm that manifests as a proxied MCP tool call. Off-MCP exfil (read a file, curl it out via a non-MCP tool) is invisible to V1 by design — it belongs to the deferred egress / fs_audit tier.
